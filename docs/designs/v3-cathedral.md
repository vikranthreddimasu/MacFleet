# CEO Plan: MacFleet v3 — The Cathedral

## Vision

### 10x Check

The current plan, fully wired, is "a working distributed training framework." The 10x version is **"the default way researchers with Apple Silicon Macs do ML, full stop."**

Concrete shape:
- `Pool().train()` genuinely works across 4+ Macs on any link type
- PyTorch-MPS AND MLX both first-class, framework-agnostic core preserved
- Cross-engine export: train in PyTorch, ship inference in MLX (one command)
- Sparse allreduce on the wire so the 200x WiFi savings claim is real
- Hybrid cloud burst: Pool(burst_to='modal') transparently scales when local fleet isn't enough
- Live TUI dashboard wired to the agent
- Security tight enough to train on untrusted WiFi safely
- Thermal-aware pause/resume so long runs don't produce NaN from throttling
- Fleet-wide LR finder as a new research primitive
- Crash-safe training via checkpoint shard streaming

### Platonic Ideal

A grad student opens a Jupyter notebook on her MacBook Air. Her partner's Mac Studio is on the apartment WiFi. Her university Mac mini is on Tailscale. She writes:

```python
import macfleet
with macfleet.Pool() as pool:
    pool.train(my_llama_lora, dataset, epochs=3)
```

Her terminal lights up with the Rich TUI: ring allreduce animation, thermal gauges, gradient magnitude histograms. World size: 3. Training starts. Her LoRA finishes in 40 minutes instead of 4 hours. She types `pool.export("mlx")` and it converts for inference. Zero cloud spend. Zero config files. Zero docs she had to read.

**Apple doesn't ship this because they focus on MLX-native users.** The PyTorch-on-Mac researcher is invisible to Apple's roadmap. MacFleet owns that gap.

## Strategic Premise (from landscape check)

- **mlx.distributed exists** and Apple is investing hard. Ring backend, pipeline and tensor parallelism. Community frameworks on top (mlx-train, DaveAldon/Distributed-ML-with-MLX). Ollama going MLX-native.
- **PyTorch-MPS has ZERO distributed path.** NCCL unsupported. Gloo broken on MPS. Single-GPU-on-MPS confirmed on PyTorch forums.
- **MacFleet's real moat:** the only distributed training path for PyTorch-on-Apple-Silicon. Dual-engine framing preserves optionality but the core positioning is "PyTorch-on-Mac distributed training, with MLX as a first-class co-engine."

## Implementation Effort (honest)

Revised after spec review: "5-7 days of focused CC work" was optimistic. Realistic:

- v2.2: ~2 weeks CC work (big because it includes security overhaul + wiring + first-ever CI)
- v2.3: ~1 week CC work (straightforward feature adds + cleanup)
- v3.0: ~2-4 weeks CC work (hybrid cloud + streaming are hard)

Ship v2.2 first. Stop. Get feedback. Then v2.3. Don't start v3.0 until v2.3 has real users.

## Risk Register

1. **Scope too big for a single person.** 8 expansions + 24 fixes is a quarter of work at human speed. With CC+gstack, v2.2 alone is ~2 weeks of focused work (see Implementation Effort below). User should batch v2.2 first and ship before starting v2.3.
2. **Compression sparse path is non-trivial.** DGC-style sparse allreduce requires careful handling of non-aligned top-k indices across peers. Reference implementation needed.
3. **MLX parity.** If E1 (cross-engine export) goes well but MLX path doesn't get same attention, we ship asymmetric quality. Must test both engines equally.
4. **Hybrid cloud (E5) is ambitious.** Deferred to v3.0 because it's an architectural step, not a feature add.
5. **The mlx.distributed question isn't going away.** If Apple ships better heterogeneous link support in MLX 1.0, MacFleet's MLX engine becomes redundant. Keep an eye on WWDC announcements.
